3.1 Forking & Cloning a Remote Repository

Cloning a repository from GitHub means creating a copy of a remote repository on your local machine. This allows you to work on the project, make changes, and later push updates back to GitHub.

Why clone a repository?

  • Work Offline - You can make changes to a project without needing access to the internet
  • Collaboration - Teams can work simulteanously on the same project by cloning the same repository
  • Backup & Version Control - You can get a full version history of the project on your local machine
  • Save Time - You can clone starter code for projects instead of setting everything up manually

But if we all want to clone the same repository but have our own copies that wont affect each others, we need to fork the repository. This will create a copy of the repository under your individual GitHub accounts. After forking the repository, we then need to clone it to ensure that any changes we push to GitHub won't affect the original repository.

Why wouldn't we initialise the repository with a README?

Whether or not to initialise a repository with a README file depends on the use case. Here's some reason why you might choose not to initialise a repository.

When to Initialise with README When NOT to Initialise with README
You are starting a new project from scratch You are pushing an existing project to the repo
You want the repo to be easily browsable on GitHub You are setting up a bare repository
The project is open-source and needs documentation from the start Your team prefers to add the README in the first meaningful commit
You want contributors to understand the project immediately The project structure requires manual file organization first

In our case, we want to push an existing project to the repository, so we won't intialise it with a README.
