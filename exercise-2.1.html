2.1 Creating a Remote Repository in GitHub

What is GitHub?

GitHub is a cloud-based platform for hosting Git repositories. It allows developers to collaborate, track changes, and manage projects.

So far, we've created a local repository on our machines, but this is only accessible by us. To make the files available for everyone working simultaneously on a project, we need to push our files to a remote repository.

Let's take a look at how to do this.

Lesson Objectives

  • Create a GitHub account
  • Create a remote repository on GitHub
  • Link our local repository to the remote repository
  • Push files and changes from the local repository to the remote repository
Step 1 of 4

Creating a GitHub Account

Set up your GitHub account by following this link - GitHub Set Up

  • Click on the ‘+’ symbol in the top right corner
  • Give your repository a name and a description
  • Set your repository to 'Private' for now as this is just an individual task
  • Do not initialise the repository with a README
  • Click 'Create repository'
Step 2 of 4

Linking a Local and Remote Repository

We're now going to link our local GitHub repository with the remote repository we've just created.

We're now going to change the state of the file by adding text to it, and then saving it in another commit.

Run the following command in Git Bash

notepad exercise_1_questions.txt

This opens the file with Notepad

Now go ahead and answer questions 1-3 on the file, save it (Ctrl + S), and then close the file.

Step 3 of 4

Stage The Recent Change

It's good practice to make frequent commits in Git at logical points of the development process.

These can be small and focused changes such as after completing a specific task, or fixing a bug.

Taking what we've learnt in the last lesson about staging a file, let's add the modified exercise_1_questions file into the staging area.

Run the following command in Git Bash

git add .

Remember that the full stop after the add command will stage all modified files.

Note: You can put a full stop after git add to specify you want all files with modifactions made to them to be tracked, rather than listing them individually

Step 4 of 5

Commit The Change

A good commit message is essential for maintaining clarity and understanding in your project, especially when working in teams or revisiting code in the future.

What makes a good commit message?

  • Clear and Descriptive: Summarises what the change is
  • Short Message: Keep it brief (60 characters or less), but provide enough context
  • Focus on a Single Change: Each commit should represent one logical change. If it doesn't, then you should probably make more than one commit

Now that we've taken a look at some best practices for commit messages, let's create one ourselves.

Run the following command in Git Bash

git commit -m"Added answers to questions 1-3 on exercise_1_questions.txt"
Step 5 of 5

Answer the Rest of The Questions and Commit the Changes

Now that we've made some changes to the file and committed these changes, go ahead and answers questions 4 and 5 on the file, and then stage and commit the file as usual.

Tip: Don't forget to save (Ctrl + S) and close the file before staging it.

Run the following commands in Git Bash

notepad exercise_1_questions.txt
git add .
git commit -m"Added answers to questions 4 and 5 on exercise_1_questions.txt"
